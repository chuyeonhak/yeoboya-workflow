1. 원래 의도한 워크플로우

  1. /work 진입점 → 작업 유형 질문 (새 기능 · 버그 수정 · 기능 강화)
  2. PDF 유무 확인
  3. PDF 있음: PDF → 스펙 변환 스킬
  4. PDF 없음: 단순 작업 판정 → Superpowers 로 스펙 작성 → Notion 업로드
  5. PDF 기반 스펙: 프로젝트 분석 → 구현 항목 / 기존 사용자 영향 / 엣지케이스 / 개발 기간 산정 → Notion 업로드
  6. 스펙 기준 Superpowers 로 PLAN 작성
  7. PLAN 기반 Superpowers TDD 로 마무리

3. 회고 — 왜 기간을 못 맞췄나

PDF 파싱 · Notion 업로드까지는 완료

의도한 7단계 중 핵심 파이프라인(PDF → 파싱 → Notion) 은 v0.3 까지 목표대로 마무리했다. Phase 1(파싱) · Phase 5(퍼블리시) 가 안정화되고, 재실행 · 재개 · 부분 갱신까지 커버.

막힌 지점 — "웹 포함 여부" 필터링

기획 PDF 에는 iOS · Android 외에 웹 전용 피처 가 섞여 있는 경우가 많아, Phase 2-6 에 "웹이 들어가는지 안 들어가는지" 를 판별하고 걸러내는 단계를 추가하려 했다. PDF 맥락에서 플랫폼을 추론하는 휴리스틱 설계 · 사용자 확인 UI · 구조화 결과에 플랫폼 태그 병합 로직이 얽혀 예상보다 커졌고, 이 기능을 넣다가 릴리스 일정을 못 맞췄다.

4. 플러그인 아키텍처

디렉터리 구조

yeoboya-workflow/
├── .claude-plugin/
│   ├── plugin.json          # 플러그인 메타 (name, version)
│   └── marketplace.json
├── commands/                # 슬래시 커맨드 엔트리포인트 (얇은 래퍼)

│   ├── spec-update.md
│   └── spec-resume.md
├── skills/
│   ├── pdf-spec-organizer/
│   │   ├── SKILL.md         # 실제 워크플로우 로직 (Phase 1~5)
│   │   ├── scripts/         # Python 파싱 · OCR · PII 스캔
│   │   └── references/      # 템플릿, 프롬프트 가이드
│   └── common/
│       └── scripts/         # pdf_hash.py, draft_registry.py 등 공용

│   │   └── plans/           # writing-plans 산출물
│   └── manual-qa.md
└── yeoboya-workflow.config.json   # 팀 공유 설정 (레포 커밋)

① 커맨드 ↔ 스킬 분리

슬래시 커맨드는 인자 파싱 · 환경변수 정규화만. 실제 워크플로우 로직은 전부 SKILL.md 안에 있다.

  • 커맨드는 "얇은 래퍼" — 교체 · 추가 비용 최소화
  • 스킬은 MODE 환경변수(update/resume)로 단일 엔트리 유지

② 팀 공유 단일 소스

yeoboya-workflow.config.json 을 레포에 커밋해 모든 팀원이 같은 Notion DB 를 바라보게 강제.

  • 없으면 즉시 중단 — 해석 분기 방지
  • 팀 리드 최초 1회 셋업 후 전원 동일 동작

③ 상태 보존 + 재개 (Resume)

초안을 /tmp/spec-draft-<hash>-<ts>/draft.md 에 저장하고, 상단의 <!-- plugin-state --> 헤더에 phase, publish_state, page_id 를 기록.

  • 중단돼도 /spec-resume 로 해당 Phase 부터 이어받기
  • Phase 5 도중 chunk append 부분 실패도 sentinel 로 복구

④ 동시 실행 감지

PDF 해시 + draft_registry.py 로 "5분 내 같은 PDF 를 누군가 실행 중" 을 감지, 덮어쓰기 전에 사용자 조율 유도.

  • iOS · Android 가 같은 PDF 를 동시에 돌려도 충돌 회피

⑤ Phase 기반 선형 파이프라인

파싱 → 구조화 → (웹 필터) → 누락 체크 → 메타 생성 → 노트 작성 → 퍼블리시. 각 Phase 는 직전 Phase 결과만 읽음.

  • 중간 실패 지점이 명확 — 재개 / 디버깅 용이
  • Phase 3.5 같은 신규 단계 삽입이 쉬움

⑥ 병합 가능 Notion 스키마

하나의 Notion 피처 페이지 안에 iOS · Android · 공통 노트 섹션이 공존. 재실행해도 상대 플랫폼 섹션을 덮어쓰지 않는다.

  • 3-way merge 로직으로 동시 편집 충돌 감지
  • 피처별 Toggle 단위 편집(--feature) 지원

5. 요약

현재 플러그인은 의도한 파이프라인의 앞단(PDF 파싱 → Notion 업로드) + 프로젝트 분석 일부 + /work 진입점 까지 구현됐다. PDF 없는 경로 는 신규 스킬 conversation-spec-organizer 의 3-Phase 설계가 끝났고 실제 배선이 진행 중. Superpowers PLAN / TDD 연계 는 여전히 미착수.

다음 마일스톤: ① conversation-spec-organizer 배선 완료 (brainstorming 호출 · draft 생성 · Notion publish 재사용) → ② 스펙 → PLAN → TDD 체인 자동화.
